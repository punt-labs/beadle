\section*{Press Release}

SEATTLE, WA --- September 15, 2026 --- Punt Labs, whose products address
the second-order effects of AI --- including the trust failures and adoption
backlash that follow when autonomous systems lack verifiable controls --- today
released Beadle, a free, open-source daemon that gives developers cryptographic
control over an autonomous agent running on their own machine. Unlike agent
frameworks that ask for broad trust and offer opaque logs, Beadle requires a
GPG-signed instruction for every action, enforces declared permissions before
executing anything, and records every step in a tamperproof, signed audit trail.
The owner triggers pipelines via CLI, email through Proton Mail, or scheduled
cron --- Beadle executes multi-step workflows using bash, Claude, or Python
regardless of how the task arrives (see \faqref{faq:competitors}).

\prsection{Problem}

Developers today use AI coding assistants interactively: they sit at the
keyboard, type a prompt, review the output, and iterate. When the laptop closes,
the agent stops. A developer who triggers a two-hour test suite at 6pm must
either keep the terminal open until 8pm or accept that a crash at 7pm means
starting over in the morning. Delegating multi-step tasks --- run tests across
three environments, summarize the results, file issues for failures --- means
cobbling together fragile cron jobs and shell scripts with no authentication, no
permissions model, and no audit trail. Existing autonomous agent frameworks like Auto-GPT run locally but offer
no cryptographic verification of who authorized which action, no per-command
permissions, and no tamperproof record of what happened.
The risk is not theoretical: in January 2026, ClawdBot (OpenClaw) gained
85,000+ GitHub stars in its first week and was immediately found to have
critical vulnerabilities --- authentication disabled by default, a skill
directory riddled with malicious extensions, and prompt injection paths that
exfiltrated API keys \cite{kaspersky2026_openclaw, snyk2026_clawdbot}.
Autonomous agents without enforceable trust boundaries produce real security
failures at scale (see \faqref{faq:customer-evidence}).

\prsection{Solution}

Beadle is a background daemon configured for a single identity: a name, an email
address, and a GPG keychain. The owner writes command documents --- small
Markdown files that declare their interpreter (bash, Claude, or Python), their
inputs and outputs, and their required permissions. Commands are signed with the
owner's GPG key and can be piped together like Unix commands:

{\small\texttt{check-schedule.md | find-free-slot.md | reserve-slot.md | notify-attendees.md}}

The owner triggers a pipeline from the CLI, by emailing Beadle, or by
scheduling it on Beadle's internal cron. Regardless of how the task arrives,
Beadle validates every signature, preflights all permissions before executing
anything, and runs the pipeline atomically. Every action is recorded in a
signed, plaintext audit log that only the owner can clear
(see \faqref{faq:why-email}).

\prsection{Getting Started}

Getting started with Beadle takes three steps. First, run
{\small\texttt{beadle init}} --- Beadle generates a GPG signing key, creates
your identity file, and sets up the audit log. Second, write your first task in
a Markdown file and sign it with {\small\texttt{beadle sign my-task.md}}.
Third, run {\small\texttt{beadle run my-task.md}} and see the result. Within
minutes, Beadle validates the signature, checks permissions, executes the
command, and logs every step with a cryptographic proof chain. No cloud account,
no data leaves your machine. To add email as a remote trigger, run
{\small\texttt{beadle init {-}{-}email}} to connect Proton Bridge --- but
the core value works without it.

\subsection*{External FAQs}

\begin{faqpair}{What is Beadle and who is it for?}\label{faq:what-is-it}
  Beadle is an open-source autonomous agent daemon for developers and power users
  who want to delegate tasks to an AI agent that runs on their own machine.
  It is designed for people who already use AI coding assistants but need a way to
  run multi-step pipelines unattended --- overnight test suites, scheduled code
  reviews, automated triage of incoming emails --- with cryptographic proof of
  what was authorized and what happened.
\end{faqpair}

\begin{faqpair}{How is Beadle different from Auto-GPT or Claude Code?}\label{faq:competitors}
  Auto-GPT and similar frameworks give an agent broad autonomy to decide its own
  next steps, with minimal authentication and coarse-grained permissions. Claude
  Code is an interactive assistant that requires the developer at the keyboard.

  Beadle occupies a different point in the design space: it has \textit{zero}
  independent authority. Every action requires a GPG-signed instruction from the
  owner. Commands declare their permissions (read, write, execute) per document.
  Pipelines are composed explicitly, not decided by the agent. The audit log is
  signed and tamperproof. The trade-off is deliberate: Beadle sacrifices agent
  autonomy for owner control and auditability.
\end{faqpair}

\begin{faqpair}{Why email? Why not a web UI or a CLI?}\label{faq:why-email}
  Email is asynchronous, universal, and already authenticated. Proton Mail
  provides end-to-end encryption. GPG provides sender verification. Together,
  they give Beadle a control plane that works from any device --- phone, tablet,
  another computer --- without installing a client or opening a port. The
  concept of email as an agent control channel has engineering precedent:
  HumanLayer's Agent Control Plane uses email for human-approval of autonomous
  agent actions \cite{humanlayer2024acp}. The owner
  can CC Beadle on a conversation with a colleague, and Beadle can distinguish
  the owner's signed instructions from the colleague's unsigned replies.

  Beadle also supports local execution via the CLI
  ({\small\texttt{beadle run pipeline.md}}) for development and testing.
  Email is the remote interface, not the only interface.
\end{faqpair}

\begin{faqpair}{How do I get started?}\label{faq:getting-started}
  Install Beadle via pip ({\small\texttt{pip install punt-beadle}}), run
  {\small\texttt{beadle init}} to configure your identity and Proton Bridge
  connection, and sign your first command with
  {\small\texttt{beadle sign my-task.md}}. Email it to your Beadle address
  or run it locally. Results arrive in your inbox within minutes.
\end{faqpair}

\begin{faqpair}{How much does it cost?}\label{faq:pricing}
  Beadle is free and open-source. You need a Proton Mail paid plan
  (starting at \$3.99/month) because Proton Bridge --- the local IMAP/SMTP
  server Beadle uses for email --- is not available on the free tier
  \cite{protonbridge2025}. You also need a GPG keychain (free). If your command
  pipelines use the Claude interpreter, you will need an Anthropic API key with
  usage-based billing. Beadle itself has no cost.
\end{faqpair}

\begin{faqpair}{What happens to my data?}\label{faq:data}
  Nothing leaves your machine unless a command document explicitly sends it
  somewhere. Beadle runs locally as a daemon. Email flows through Proton Bridge
  on your machine to Proton's encrypted servers. Command documents, audit logs,
  and results stay on your local filesystem. There is no telemetry, no cloud
  service, and no account with Punt Labs.
\end{faqpair}

\begin{faqpair}{Do I need to know GPG?}\label{faq:gpg-knowledge}
  No. {\small\texttt{beadle init}} generates a signing key in an isolated
  keychain (separate from your system GPG keyring), configures Proton Bridge,
  and produces a single identity file. You never run \texttt{gpg} directly ---
  {\small\texttt{beadle sign}} and {\small\texttt{beadle verify}} handle all
  cryptographic operations. If you already have a GPG key you want to use,
  Beadle can import it, but it still stores it in its own keychain. The only
  requirement is that signing keys must have an expiration date ---
  Beadle enforces this and will not accept non-expiring keys for command
  signing.
\end{faqpair}

\subsubsection*{Value \& Market}

\begin{faqpair}{What is the total addressable market?}\label{faq:tam}
  The primary market is developers who already use AI coding assistants and want
  unattended execution. The LangChain 2024 State of AI Agents report surveyed
  1,300+ professionals and found that 51\% are using AI agents in production
  today, with 78\% planning to deploy agents soon \cite{langchain2024agents}.
  The subset who need local-first, authenticated, autonomous execution is
  narrower --- likely tens of thousands of developers who work with sensitive
  codebases, regulated environments, or privacy-conscious infrastructure
  (the Proton Mail user base is an imperfect proxy for this audience).

  This is a bottoms-up estimate. We are not claiming a billion-dollar TAM.
  The honest framing: Beadle targets a niche of security-conscious developers
  who value auditability over convenience.
\end{faqpair}

\begin{faqpair}{What evidence do we have that customers want this?}\label{faq:customer-evidence}
  At hypothesis stage, we have limited primary data but one working prototype.
  The evidence:

  \textbf{Primary evidence (owner use):} A proof-of-concept runs daily on the
  author's machine. A GPG-signed prompt policy manifest
  ({\small\texttt{prompts.policy.yml}}) defines a healthcheck command with
  SHA-256 integrity verification, trusted key enforcement, per-command
  constraints (model, max cost, allowed tools, I/O boundaries), and template
  variable substitution. A shell wrapper verifies the GPG signature, invokes
  Claude non-interactively, and delivers results by email via Proton Bridge.
  The agent has its own Proton Mail address and delivers results via Proton
  Bridge. This prototype has run reliably as a daily LaunchAgent for several
  weeks, validating the core integration: GPG-signed prompts, Claude CLI
  invocation, Proton Bridge email delivery, and scheduled execution.

  \textbf{Inferential evidence:}
  \begin{itemize}[nosep]
    \item Auto-GPT has accumulated over 182,000 GitHub stars since its 2023
      release, demonstrating strong developer interest in autonomous agents
      \cite{autogpt2024github}.
    \item The OWASP GenAI Security Project published a Top~10 for Agentic
      Applications in 2025, indicating that agent security is a recognized
      concern, not a fringe worry \cite{owasp2025agentic}.
    \item Proton Mail has grown to 100M+ accounts, confirming demand for
      privacy-focused communication infrastructure \cite{proton2024100m}.
    \item Cisco's 2025 Data Privacy Benchmark found that 90\% of
      organizations see local storage as inherently safer, and 64\% worry
      about exposing sensitive data to cloud GenAI tools
      \cite{cisco2025privacy}.
    \item No existing agent framework offers GPG-based authentication with
      per-command permissions --- the gap is observable, but unvalidated.
  \end{itemize}

  \textbf{Next validation step:} Generalize the prototype into the Beadle
  daemon (multi-command pipelines, identity management via
  {\small\texttt{beadle init}}). Then test with 5--10 external developers
  who currently use Auto-GPT or Claude Code for automation tasks. Measure
  whether the GPG authentication model feels like a feature or a burden.
\end{faqpair}

\begin{faqpair}{Who are the competitors and why will we win?}\label{faq:competitive-landscape}
  \begin{tabularx}{\linewidth}{@{} l X X @{}}
  \toprule
  \textbf{Competitor} & \textbf{Strength} & \textbf{Trust model gap} \\
  \midrule
  Auto-GPT & Large community, broad agent autonomy &
    Agent-directed execution; no owner authentication, no declared
    permissions, no tamperproof audit trail \\
  Claude Code & High-quality code generation, interactive &
    Operator-attended trust model; requires user at keyboard,
    cannot delegate unattended authority \\
  Devin & Full autonomous dev environment &
    Vendor-hosted trust model; execution runs on Cognition's
    infrastructure, not the owner's machine \\
  Cron + scripts & Simple, well-understood &
    No trust model; no identity verification, no permission
    scoping, no composability with LLMs \\
  \bottomrule
  \end{tabularx}

  \vspace{0.5em}
  Beadle does not compete on agent intelligence --- it uses existing LLMs
  (Claude, GPT) as interpreters. The differentiation is the trust model:
  local-only execution under cryptographic owner control. Cloud providers and
  SaaS agent platforms cannot replicate this without contradicting their own
  business model --- their revenue depends on hosting execution, managing
  infrastructure, and retaining custody of the runtime environment. Offering
  genuine local-only, owner-controlled execution means giving up the recurring
  billing, telemetry, and platform lock-in that sustain their economics.
  Competitors can add daemon modes, unattended execution, and even signing
  features, but they cannot cede control of the execution environment to the
  owner without dismantling the value capture that funds their products. This
  is a structural misalignment, not a feature gap.
\end{faqpair}

  The proof-of-concept already validates three core mechanics: GPG-signed

  These run reliably as a daily LaunchAgent on the author's machine.

\end{faqpair}

\subsubsection*{Technical}

\begin{faqpair}{What are the major technical risks?}\label{faq:tech-risks}
  \begin{enumerate}[nosep]
    \item \textbf{Proton Bridge reliability for automated use.} Proton Bridge
      is designed for desktop email clients, not daemons. It may not handle
      long-running, unattended operation gracefully.

      \textit{Why Proton specifically:} Beadle's threat model assumes the
      email channel must protect command integrity end-to-end --- from the
      owner's device to the daemon's mailbox. Proton Mail's end-to-end
      encryption means even Proton's own servers cannot read command content in
      transit, which aligns with the local-first, zero-trust design. That said,
      Beadle connects to Proton Bridge via standard IMAP/SMTP --- there is no
      Proton-specific API or protocol dependency. Supporting other email
      providers (Fastmail, self-hosted Dovecot, Gmail with app passwords) is a
      scope decision, not an architectural constraint.

      \textit{Mitigation:} Beadle monitors Bridge health on a configurable
      interval (default: 60 seconds) and exposes status via
      {\small\texttt{beadle status}}. If Bridge becomes unavailable, Beadle
      logs the outage, queues pending result deliveries, and retries on a
      backoff schedule. On macOS, the LaunchAgent plist includes a
      {\small\texttt{KeepAlive}} directive that restarts the daemon
      automatically if it exits; on Linux, the systemd unit specifies
      {\small\texttt{Restart=on-failure}} with a 30-second delay. The owner
      can configure alerting (e.g., a local notification or a write to a
      monitored file) when Bridge has been offline for longer than a threshold.
      Email-triggered execution is unavailable while Bridge is down, but
      locally triggered and cron-triggered pipelines continue to run and their
      results are delivered once Bridge recovers.
    \item \textbf{GPG key management complexity.} GPG tooling is notoriously
      difficult --- complexity is consistently cited as its primary operational
      weakness \cite{hoopdotdev2024gpg}. If {\small\texttt{beadle init}} cannot
      fully abstract key generation and signing, the onboarding friction will
      kill adoption. This is the hardest unsolved UX problem in the project.

      \textit{Alternative evaluated: SSH signing.} GitHub and GitLab both
      support SSH keys for commit signing, and most developers already have an
      SSH key. However, SSH signing only covers commit-level attestation ---
      it provides no standard for encrypting documents to a recipient, no
      built-in key expiry or revocation model, and no ecosystem for
      distributing trust (no equivalent of the OpenPGP Web of Trust or
      keyservers). Beadle needs signing \textit{and} encryption \textit{and}
      key lifecycle management; SSH covers only the first.

      \textit{Mitigation:} {\small\texttt{beadle init}} must make GPG
      invisible. Concretely, it must: (1)~generate an Ed25519 signing key
      with a mandatory expiration date, storing it in an isolated
      {\small\texttt{GNUPGHOME}} so it does not pollute the user's existing
      keychain; (2)~configure Proton Bridge IMAP/SMTP credentials and verify
      connectivity with a round-trip test message; (3)~produce a single
      {\small\texttt{beadle.yml}} identity file that all subsequent commands
      consume. The user should never run \texttt{gpg} directly.
      {\small\texttt{beadle sign}} wraps signing, and
      {\small\texttt{beadle verify}} wraps verification. If this abstraction
      leaks --- if any error message surfaces raw \texttt{gpg} output or
      requires the user to understand keyring semantics --- the onboarding
      has failed.
    \item \textbf{Claude interpreter sandboxing.} When Beadle invokes Claude
      as a command interpreter, the LLM may attempt actions outside the
      command's declared permissions. \textit{Mitigation:} The two-level
      try-catch model preflights all permissions before execution. The Claude
      interpreter runs in a restricted context with only the declared
      inputs available.
  \end{enumerate}
\end{faqpair}

\begin{faqpair}{What dependencies exist on other teams or systems?}
  \begin{itemize}[nosep]
    \item \textbf{Proton Bridge} --- open-source Go codebase, maintained by
      Proton AG \cite{protonbridge2025}. Beadle uses it as a standard
      IMAP/SMTP client. No API dependency beyond the mail protocol.
    \item \textbf{Anthropic API} --- required only when using the Claude
      interpreter. Usage-based billing, no contractual dependency.
    \item \textbf{GnuPG} --- the GPG implementation. Mature, widely deployed,
      maintained by the GnuPG Project. RFC~9580 (OpenPGP, 2024) provides
      the current standard \cite{rfc9580}.
  \end{itemize}
  No internal Punt Labs dependencies. Beadle is a standalone project.
\end{faqpair}

\begin{faqpair}{What is the scaling story?}\label{faq:scaling}
  Beadle is a single-user daemon. ``Scaling'' means long-running operation over
  months and years, not multi-tenant growth.

  \begin{itemize}[nosep]
    \item \textbf{Audit log growth.} The log is append-only and unbounded until
      the owner manually clears it. A developer running 10 pipelines per day
      generates roughly 50--100~KB of log per day. At one year, that is
      approximately 20--40~MB --- manageable on any modern filesystem. The risk
      is neglect, not size.
    \item \textbf{Command library size.} Commands are loaded into memory only
      during preflight and execution, then discarded. A library of hundreds of
      signed command documents on disk has no runtime cost until invoked.
    \item \textbf{Proton Bridge longevity.} The daemon depends on Bridge staying
      online for email-triggered execution. Bridge was not designed for months
      of continuous uptime. The mitigation is operational: Beadle checks Bridge
      health on a configurable interval, the system service manager (LaunchAgent
      or systemd) restarts both Beadle and Bridge on failure, and pending result
      deliveries are queued and retried once Bridge recovers. Local and
      cron-triggered pipelines are unaffected by Bridge downtime.
  \end{itemize}

  The honest constraint: Beadle does not scale to teams, organizations, or
  multi-machine deployment. It is a personal agent for one person on one
  machine. That is a design choice, not a limitation to fix later.
\end{faqpair}

\subsubsection*{Business}

\begin{faqpair}{What is the revenue model?}\label{faq:revenue}
  None. Beadle is free and open-source. The strategic value is threefold:
  (1)~it demonstrates the Punt Labs engineering methodology in a real,
  useful product; (2)~it grows the Punt Labs open-source ecosystem alongside
  Biff, Quarry, and TTS; (3)~it serves as the reference implementation for
  a future punt-labs project on use-case methodology, generating content and
  credibility.
\end{faqpair}

  \vspace{0.5em}
  \textbf{Pre-mortem --- what failure looks like:} It is March 2027. Beadle has
  200 GitHub stars, 8 PyPI installs per week, and the prototype test showed
  3/10 users abandoned during GPG key setup. The cryptographic trust model ---
  signed commands, declared permissions, tamperproof audit trail --- resonated
  in blog posts but not in practice. Developers found the signing ceremony
  too heavy for personal automation tasks where they already trust themselves.
  The security posture Beadle enforces felt like overhead, not protection.
  The project becomes a methodology showcase only, with no active user
  community, and enters archival maintenance.
\end{faqpair}

\begin{faqpair}{Why now? What has changed?}\label{faq:why-now}
  Four shifts make this the right time:

  \begin{enumerate}[nosep]
    \item \textbf{AI agents went mainstream in 2024--2025.} The LangChain survey
      shows 51\% production deployment \cite{langchain2024agents}. Developers
      are past the question of ``should I use an agent?'' and into ``how do I
      trust one?''
    \item \textbf{OpenPGP modernized.} RFC~9580 (2024) updated the OpenPGP
      standard for the first time in decades, adding modern cryptographic
      algorithms and improving key management \cite{rfc9580}. GPG tooling is
      better than it was even two years ago.
    \item \textbf{OWASP formalized agent security risks.} The OWASP Top~10 for
      Agentic Applications (2025) legitimized the concern that autonomous agents
      need security models, not just capability models
      \cite{owasp2025agentic}. Beadle's trust model is a direct response.
    \item \textbf{ClawdBot proved the threat model in production.} In January
      2026, ClawdBot (also known as OpenClaw) launched and gained 85,000+
      GitHub stars in its first week. Within days, security researchers
      discovered critical vulnerabilities: CVE-2026-25253 (CVSS~8.8) allowed
      gateway compromise via authentication bypass in misconfigured reverse
      proxies, CVE-2026-24763 enabled hundreds of malicious skills in the skill
      directory, and CVE-2026-25157 allowed prompt injection to extract secrets
      \cite{kaspersky2026_openclaw}. Snyk demonstrated that a spoofed email
      could trigger a shell command that exfiltrated \texttt{clawdbot.json}
      containing API keys \cite{snyk2026_clawdbot}. Palo Alto Networks'
      Unit~42 concluded that OpenClaw ``maps to all 10 OWASP risks for agentic
      applications'' and lacked ``enforceable trust boundaries between untrusted
      inputs and high-privilege reasoning'' \cite{unit42_2026_openclaw}.
      Kaspersky reported that authentication was disabled by default
      \cite{kaspersky2026_openclaw}. This is the incident that proved
      autonomous agents without cryptographic trust models are dangerous at
      scale --- exactly the failure mode Beadle is designed to prevent.
  \end{enumerate}
\end{faqpair}

\section*{Feature Appendix}

Defines the scope boundary for the product. Every feature is explicitly
categorized to prevent scope creep and force prioritization decisions upfront.

\subsection*{Must Do}

Features that are essential for launch. Without these, the product does not
solve the core problem.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Identity configuration}{single-identity setup with name, email,
    and GPG keychain via {\small\texttt{beadle init}}}\label{feat:identity}
  \featureitem{GPG command signing}{sign command documents with owner's key;
    reject unsigned or expired-key commands}\label{feat:signing}
  \featureitem{Command pipeline execution}{pipe commands together with
    Unix-style syntax; two-level try-catch (preflight then
    execute)}\label{feat:pipelines}
  \featureitem{Three interpreters}{bash, Claude, and Python as command
    interpreters, declared per document}\label{feat:interpreters}
  \featureitem{Per-command permissions}{rwx permission bits per command
    document, validated at preflight}\label{feat:permissions}
  \featureitem{Signed audit log}{plaintext, GPG-signed log of every action;
    only owner can clear}\label{feat:audit-log}
  \featureitem{Email trigger via Proton Bridge}{receive GPG-signed instructions
    by email and return results by email}\label{feat:email}
  \featureitem{Local CLI execution}{run pipelines locally via
    {\small\texttt{beadle run}} for development and
    testing}\label{feat:local-cli}
\end{enumerate}

\subsection*{Should Do}

Features that meaningfully improve the product but are not launch-blocking.
Candidates for fast follow-up.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Internal scheduler}{cron-like scheduled pipeline execution with
    no catch-up on missed runs}\label{feat:scheduler}
  \featureitem{Third-party email handling}{quarantine and escape unsigned
    third-party content from forwarded emails}\label{feat:third-party}
  \featureitem{Key rotation management}{guided key rotation with expiry
    enforcement; remove keys from trusted list}\label{feat:key-rotation}
  \featureitem{Command schema validation}{declare input/output schemas per
    command; preflight validates compatibility between adjacent pipeline
    stages}\label{feat:schemas}
  \featureitem{Multi-provider email support}{Beadle speaks standard
    IMAP/SMTP --- supporting other providers (Fastmail, Dovecot, Gmail)
    is a scope decision, not an architectural lock-in. Proton-only in
    Phase~0 because E2E encryption aligns with the trust
    model}\label{feat:multi-provider}
\end{enumerate}

\subsection*{Won't Do}

Features explicitly excluded. Naming what you won't build prevents scope
creep and clarifies the product's identity.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Multi-identity support}{Beadle is a personal agent for one
    owner. Multi-tenant adds complexity that conflicts with the single-identity
    trust model}\label{feat:no-multi-identity}
  \featureitem{Signal or other messaging integrations}{email is the remote
    interface. Adding messaging platforms fragments the authentication model
    and increases attack surface}\label{feat:no-signal}
  \featureitem{Web UI}{a browser interface contradicts the local-first,
    CLI-native design philosophy and adds a large attack
    surface}\label{feat:no-web-ui}
  \featureitem{Cloud hosting or SaaS model}{Beadle runs on the owner's machine.
    A hosted version would undermine the core value proposition of data
    sovereignty and local control}\label{feat:no-cloud}
  \featureitem{Scheduler catch-up}{missed scheduled runs are skipped, not
    replayed. Catch-up logic adds complexity and ambiguity about which version
    of a command should execute}\label{feat:no-catchup}
  \featureitem{Non-expiring command-signing keys}{all command-signing keys must
    have an expiration date. This is a security invariant, not a
    convenience trade-off}\label{feat:no-nonexpiring}
\end{enumerate}
